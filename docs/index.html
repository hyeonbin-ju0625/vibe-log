07 · ADR 상세 (1/3)

왜 Flutter인가?

ADR-0001
모바일 프레임워크: FlutterAccepted
고려한 대안:
React Native, Kotlin(Android 네이티브), Swift(iOS 네이티브)
결정 이유:
코드 한 벌로 멀티 플랫폼 지원, UI 위젯이 풍부해 디자인 시스템 빠르게 구축, 초급자도 학습 곡선이 완만함.
트레이드오프:
Dart 언어를 새로 배워야 하지만 문법이 직관적이라 부담 적음.
07 · ADR 상세 (2/3)

왜 Provider인가?

ADR-0002
상태관리: ProviderAccepted
고려한 대안:
Riverpod, Bloc, GetX
결정 이유:
Flutter 공식 권장 방식이며 학습 곡선이 가장 낮음. 현재 앱 규모에서 충분히 적합.
트레이드오프:
앱이 매우 커지면 Riverpod 전환이 필요할 수 있음.
07 · ADR 상세 (3/3)

왜 Firebase인가?

ADR-0003
백엔드: FirebaseAccepted
고려한 대안:
REST API 직접 구축, 로컬 SQLite
결정 이유:
별도 서버 개발 없이 빠르게 구축, 인증·DB·스토리지 통합 제공, 무료 할당량으로 학생 프로젝트에 충분.
트레이드오프:
Google 생태계 의존성, 무료 한도 초과 시 비용 발생.
08 · 구현 시행착오

인증·API 문제 해결

⚠️ Gemini API 인증 실패 (401 UNAUTHENTICATED)
문제새로 발급된 AQ. 접두사 키가 OAuth 형식으로 인식되어 표준 generativelanguage 엔드포인트에서 거부됨.
해결서비스 계정 생성 → 편집자 권한 부여 → API 키를 서비스 계정에 바인딩. 인증 헤더를 x-goog-api-key로 변경.
⚠️ API 호출 시 429/503 빈번 발생
문제무료 티어 분당 요청 제한 + 서버 과부하로 사용자가 다시 시도해야 하는 UX 발생.
해결자동 재시도 로직 추가 (503은 점진적 백오프, 429는 35초 대기). 최대 3회까지 자동 재시도하여 사용자 부담 제거.
